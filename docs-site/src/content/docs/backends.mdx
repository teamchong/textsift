---
title: Backends
description: Two execution engines behind the same public API.
---

textsift ships two interchangeable backends. The public API (`PrivacyFilter.create`, `redact`, `detect`) is identical across both — the backend is an implementation detail picked at construction.

| Backend | Compute |
|---|---|
| `webgpu` | Custom WGSL kernels (int4 matmul, banded attention, sparse MoE). Requires `shader-f16`. |
| `wasm` | Custom Zig + SIMD128 → WASM. Multi-thread when COOP/COEP headers set. |

## Backend decision tree

```
 PrivacyFilter.create({ backend?: ... })
              │
              ▼
 ┌────────────────────────────────────────┐
 │ explicit backend ?                     │
 │   "webgpu" → use it                    │
 │   "wasm"   → use it                    │
 └────────────┬───────────────────────────┘
              │ (no explicit backend or "auto")
              ▼
 ┌────────────────────────────────────────┐
 │ navigator.gpu + shader-f16 available ? │
 └────────────┬───────────────────────────┘
       yes    │    no
       ▼            ▼
   backend:"webgpu"   backend:"wasm"
```

## Compatibility

| | Chromium 147 | Firefox 129 | Safari 19 | Node 22 |
|---|:-:|:-:|:-:|:-:|
| `webgpu` | ✅ | ⚠️ `shader-f16` preview | ⚠️ `shader-f16` preview | ❌ |
| `wasm` | ✅ | ✅ | ✅ | ✅ |

`backend: "wasm"` is the only currently-working CPU path for `model_q4f16.onnx` in browsers. transformers.js with `device: "wasm"` fails session creation because ORT-Web's WASM bundle has no implementation for the `GatherBlockQuantized` and `MatMulNBits` ONNX contrib ops the int4 export uses.

For Node-native execution (no WASM, full SIMD width), the `textsift` bare-import path is reserved for the upcoming NAPI binding (issue #79). The high-level `PrivacyFilter` still throws — use `textsift/browser` for Node-via-WASM. The underlying Metal-direct kernel layer is done on macOS (all 15 kernels byte-equal vs browser, ~1.9× faster end-to-end at T=32; see `packages/textsift/src/native/STATUS.md`). Linux Vulkan-direct is the next backend.

## Output parity

Both backends produce **byte-identical spans** on the same input (`tests/browser/smoke.spec.ts` + `webgpu-smoke.spec.ts`). Logit magnitudes drift by up to ~0.2 RMS due to fp16 rounding differences accumulating across 8 transformer layers, but argmax (and hence Viterbi + span decode) is preserved.

## Switching backends at runtime

A `PrivacyFilter` instance is bound to one backend for its lifetime. To switch, dispose and recreate:

```ts
const wasm = await PrivacyFilter.create({ backend: "wasm" });
await wasm.redact(text);
wasm.dispose();

const gpu = await PrivacyFilter.create({ backend: "webgpu" });
```

Both share the same OPFS cache for the 770 MB model — switching is a ~1-second warmup, not a full re-download.

## Memory footprint

| Backend | Resident memory during inference |
|---|---|
| `webgpu` | ~30 MB JS heap + ~800 MB GPU buffers |
| `wasm` | ~800 MB WASM linear memory |

`webgpu` is the most memory-efficient since weights live in GPU buffers and JS holds only kernel-dispatch state.
