All numbers measured on an M3 Pro MacBook, Chromium 147 with `--enable-unsafe-webgpu --use-angle=metal`. Median of 5 samples after 2 warmup iterations. Reproduce with `tests/browser/bench.spec.ts`. Baseline (`tjs`) is forced via `createWithResolver` so the umbrella's auto-WebGPU short-circuit can't accidentally measure WebGPU twice.

These numbers will look different on your hardware. They're a snapshot, not a guarantee.

## Per-forward latency

| Input length | textsift (WebGPU) | textsift (WASM MT) | tjs (WebGPU default) |
|---|---:|---:|---:|
| ~7 tokens | 7.6 ms | 28.5 ms | 30.8 ms |
| ~25 tokens | 11.6 ms | 46.7 ms | 38.4 ms |
| ~80 tokens | 22.2 ms | 97.9 ms | 56.3 ms |

Both WebGPU paths are GPU-bound at this model size; the differential reflects the kernels (textsift's hand-tuned int4 matmul / banded attention vs ORT-Web's int4 implementation via WebGPU contrib ops). WASM-MT (6-worker pool, COOP/COEP headers) keeps pace with tjs on short inputs and falls behind by ~1.7× on the 80-token sample, where ORT-Web's WebGPU dispatch wins. Single-thread WASM is roughly 4× slower than MT.

## Cold start

What we measure: time from `PrivacyFilter.create()` to the model being ready for inference, **after a separate untimed prelude that pre-populates every backend's persistent cache**. The 770 MB model download is excluded — it's a CDN test, not a backend benchmark.

| Backend | Cold start |
|---|---:|
| tjs (Cache API default) | 14.5 s |
| textsift WASM (OPFS default) | 2.1 s |
| textsift WebGPU (OPFS default) | 2.4 s |

**textsift is ~7× faster cold than transformers.js in default config.**

The gap isn't ONNX parse or GPU pipeline compile — those are negligible (see breakdown below). It's storage. transformers.js's default cache is the [Cache API](https://developer.mozilla.org/en-US/docs/Web/API/Cache), which has per-resource size limits that the 770 MB model exceeds (`QuotaExceededError` in the bench logs). Cache API silently rejects the put, so every visit re-downloads. textsift writes to OPFS, which has no such limit, and the second-load cost is just `~770 MB sequential disk read + ONNX parse + backend-specific upload` ≈ 1–2 s.

A transformers.js user who plugs in a custom OPFS-backed adapter would close the gap. The factual claim is **"textsift caches large models persistently with no configuration"**, not "textsift's inference engine is faster at startup".

## Warmup breakdown (cache hit)

From `tests/browser/webgpu-coldstart.spec.ts`, second warmup:

| Step | Cost |
|---|---:|
| Adapter + device request | 0.20 s (Metal driver warm-up) |
| OPFS read (770 MB model) | 0.36 s |
| ONNX parse | 0.001 s |
| GPU buffer upload | 0.38 s |
| Pipeline compile (14 WGSL shaders, parallel) | 0.002 s |
| **Total** | **0.93 s** |

## Accuracy

`openai/privacy-filter` reports F1 96% on PII-Masking-300k in OpenAI's own evaluation. textsift doesn't change the model — all three backends produce byte-identical span output on the same input (`tests/browser/smoke.spec.ts`, `webgpu-smoke.spec.ts`). The detection accuracy is the model's, not ours.

The forward-parity test (`tests/browser/webgpu-forward.spec.ts`) confirms argmax agrees on 32/32 tokens of a deterministic input, with logit RMS drift of 0.18 vs the WASM reference — fp16 rounding compounding across 8 transformer layers. Argmax (and downstream Viterbi + span output) is preserved.

## Reproduce

```sh
npm install
npm run build
npx playwright test bench.spec.ts
```

First run downloads the 770 MB model to OPFS (~15 seconds), subsequent runs are cache-hit fast.
