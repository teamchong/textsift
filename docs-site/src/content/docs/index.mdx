## What this is

A npm package that runs [openai/privacy-filter](https://huggingface.co/openai/privacy-filter) — a 1.5B / 50M-active MoE bidirectional token classifier for PII detection — entirely on the user's device. WebGPU when available, WASM (Zig + SIMD128) otherwise. Apache 2.0.

The model is OpenAI's; the value of this package is packaging:

- A native o200k-style BPE tokenizer in pure TypeScript (no `@huggingface/transformers` dep on the lean install).
- Two custom backends — WGSL kernels for WebGPU and Zig-compiled SIMD WASM — that produce the same logits as transformers.js's WebGPU path on the same input.
- A WebAssembly path that works at all on this model: ORT-Web's WASM EP (which transformers.js uses under the hood) is missing the `MatMulNBits` and `GatherBlockQuantized` kernels the int4 export needs, so `transformers.js` with `device: "wasm"` fails session creation. textsift's WASM backend is a from-scratch implementation that doesn't go through ORT-Web.
- Persistent OPFS caching of the 770 MB model weights, configured by default. Same data the model needs anyway, just stored on the user's disk so the second visit doesn't re-download. transformers.js can be configured to do this too — by default it uses the Cache API which silently rejects payloads this big.
- Streaming overloads of `detect()` and `redact()` — pass an `AsyncIterable<string>` (LLM token stream, ReadableStream-derived iterable, etc.) instead of a `string`. `detect(stream)` yields spans as they become detectable; `redact(stream)` yields safe-to-emit redacted text fragments. O(N) total work over a stream of N tokens, vs O(N²) for the naive "call detect after every chunk" pattern.

## Two packages

```bash
npm install textsift-core   # 76 KB gzipped, no runtime deps
npm install textsift        # 226 KB gzipped, adds a transformers.js fallback backend
```

Both expose the same `PrivacyFilter` API. Pick the umbrella `textsift` if you need the transformers.js fallback (typically: browsers without WebGPU and without cross-origin-isolation for SIMD-WASM). Pick `textsift-core` if you don't.

## 30-second example

```ts
import { PrivacyFilter } from "textsift-core";

const filter = await PrivacyFilter.create();
const result = await filter.redact(
  "Hi, my name is John Smith and my email is john@example.com.",
);

console.log(result.redactedText);
// "Hi, my name is [private_person] and my email is [private_email]."
```

## Measured numbers

M3 Pro MacBook, Chromium 147, median of 5 forward passes after 2 warmups. Reproduce with `npx playwright test bench.spec.ts`.

| Input length | textsift (WebGPU) | textsift (WASM ST) | tjs (WebGPU default) |
|---|---:|---:|---:|
| ~12 tokens | 7 ms | 66 ms | 29 ms |
| ~25 tokens | 12 ms | 158 ms | 39 ms |
| ~80 tokens | 25 ms | 342 ms | 57 ms |
| Second-visit warmup ¹ | 1.1 s | 1.8 s | 11.2 s |

¹ With OPFS caching configured (textsift default). transformers.js's default Cache API rejects 770 MB; if you wire OPFS into transformers.js manually, its second-visit warmup is comparable to textsift's.

The forward-pass numbers reflect kernel-level differences (custom int4 WGSL vs ORT-Web's int4 WGSL via WebGPU contrib ops). They don't depend on caching configuration.

## CPU-only environments

`textsift` is the only currently-working path for `model_q4f16.onnx` outside of WebGPU. transformers.js's WASM EP can't load this model (missing `MatMulNBits` / `GatherBlockQuantized`); textsift's WASM backend is a separate implementation that runs the int4 matmul + GQA + sparse MoE in Zig-compiled SIMD128. Browsers without WebGPU, server-side Node, edge workers — all work.

## Caveats

`openai/privacy-filter` is a detection aid, not an anonymization guarantee. Read the [caveats page](/textsift/caveats/) and OpenAI's [model card](https://cdn.openai.com/pdf/c66281ed-b638-456a-8ce1-97e9f5264a90/OpenAI-Privacy-Filter-Model-Card.pdf) before treating output as compliance-safe.
