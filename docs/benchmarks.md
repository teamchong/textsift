## Notes

- "Cold" (first browser visit, full model download) is dominated by the
  772 MB ONNX download, not compute. Cache-API hit reduces the second page
  load to the compile + warmup cost (~1ŌĆō2 s on M-series).
- `int4` and `int8` both map to `model_q4f16.onnx` today. See
  `src/js/backends/transformers-js.ts::dtypeFor` for the rationale.
- `fp16` maps to `model_fp16.onnx` (~2 GB). Only use when memory permits and
  you want maximum accuracy.

## Stage 1 ŌĆö Zig+WASM (Node CPU bench)

Warm median per forward, measured with `npm run bench:e2e` against the full
944 MB int4-experts blob generated by `gen-full-parity-fixture.py`.

### Apple Silicon M-series, Node 20 (no WebGPU, no BLAS)

| Tokens | transformers.js (ORT-node CPU EP + BLAS) | Zig+WASM (Stage 1) |
|-------:|-----------------------------------------:|-------------------:|
|      8 |                                    50 ms |              73 ms |
|     25 |                                    43 ms |             202 ms |
|     61 |                                    61 ms |             451 ms |

**Caveat: onnxruntime-node CPU EP uses native BLAS (Accelerate/MKL/OpenBLAS)
and is multi-threaded.** Our Zig+WASM is single-threaded SIMD128. This comparison
is apples-to-oranges ŌĆö the Node CPU EP is NOT the "transformers.js WASM" path we
targeted for Stage 1. The relevant comparison (ORT Web's WASM fallback in a
browser, no WebGPU available) is pending a browser benchmark harness.

Argmax class agreement on the sample input is **41/41 identical to
transformers.js**, so the decision path is bit-equivalent in practice.

### What we measured internally

Peak kernel throughput on the same hardware:
- `matmul_bf16` (4 accumulators, 16-wide unroll): **~17 GFLOPS** across
  Phase D shapes (classifier head, router, K/V, Q, O projections).
- `matmul_bf16_x_int4block` (TR=4 tiled, wide-decode): **~13 GFLOPS**.
- End-to-end 41-token warm forward: **321 ms** (blob + warmup amortized).

### Remaining headroom

- WASM SIMD128 peaks at ~25% of native peak on modern Apple cores. The
  dominant matmul path is already in the 60-70% of WASM-SIMD-128 peak
  window (estimated); further single-threaded gains would come from
  `@mulAdd` (relaxed_madd), which trades bit-exact parity for ~1.5ŌĆō2├Ś.
- WebGPU backend (Stage 2, deferred) would skip CPU entirely.
- Multi-threaded WASM (SharedArrayBuffer + Atomics) isn't wired yet; the
  kernels are thread-safe by construction (pure + output-disjoint), so
  layer-parallel or expert-parallel scheduling is a future option.
